title: 事件与扩展点
description: 如何通过领域事件、监听器与服务边界接入自定义行为。
---

如果你想以干净方式扩展系统，而不是在核心流程里直接打补丁，先读这页。

## 代码库中的事件领域

应用为以下业务域提供了独立事件目录：

- balance
- order
- referral
- subscription
- user

监听器在 `app/Listeners` 下按相同业务域分组。

## 事件监听器应承担什么

适合放在监听器中的职责：

- 发送通知
- 触发支付后副作用
- 同步外部系统
- 埋点分析或记账动作
- 业务资源的开通或回收

不适合放在监听器中的职责：

- 核心校验逻辑
- 主事务控制流程
- 为保证数据一致性而必须同步执行的业务规则

## 实践示例

常见的事件驱动扩展点包括：

- 用户注册
- 订阅创建、续费或取消
- 订单完成
- 支付失败
- 用户验证

这可以作为你判断“该从哪里扩展产品”的思维模型。

## 推荐扩展顺序

新增自定义行为时，建议按以下顺序评估：

1. 扩展现有 service 方法
2. 使用事件监听器
3. 替换提供商实现或新增提供商
4. 最后手段才是覆盖 controller

这样更能保证自定义实现与应用其余部分保持一致。

## 其他可扩展边界

除事件外，ADMIN9 还通过以下位置提供了清晰边界：

- `app/Services`
- `app/Services/PaymentProviders`
- `app/Services/VerificationProviders`
- Filament 资源与页面
- Livewire 组件

## 建议的文档习惯

每次引入新事件或 Hook 时：

- 用业务动作命名
- 记录其触发时机
- 记录保证提供的 payload
- 记录其在持久化之前还是之后触发

这套纪律会让后续集成明显更容易。
